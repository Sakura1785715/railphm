### Task 17 已完成：健康度与告警映射规则
- 已在 `railphm-server` 中正式固化“风险分数 -> 健康度 -> 告警级别”的最小业务规则闭环。
- 当前健康度计算规则已明确为：
  - `health_score = 100 - risk_score * 100`
- 当前告警级别映射规则已明确为：
  - `0 <= health_score <= 30`：`HIGH`
  - `30 < health_score <= 70`：`MEDIUM`
  - `70 < health_score <= 100`：`LOW`
- 当前阶段上述规则统一放在后端业务解释层，而不是放在 AI 服务中，确保未来即使替换为真实神经网络或引入概率校准，业务规则层仍可保持稳定。
- 当前 `POST /api/v1/predictions/infer` 的最终输出中，后端会基于 `risk_score` 重新计算：
  - `health_score`
  - `alert_level`
- 即使 AI mock 服务返回了 `health_score` 或 `alert_level`，当前后端仍以后端规则计算结果为准，避免业务语义漂移。
- 当前已形成稳定的最小业务闭环：
  - 推理结果
  - 健康度
  - 告警级别
- 已补充自动化测试，覆盖：
  - 高风险、中风险、低风险对应不同健康度与告警级别
  - `30`、`70` 等关键边界值行为
  - 后端规则覆盖 AI 返回中的同名字段
  - 异常风险值的兜底处理与规则稳定性

## 当前目录建设进展
当前后端已逐步形成以下核心基础模块：
- `run.py`：服务启动入口
- `app/core/`：响应封装、异常处理、日志等核心能力
- `app/extensions/`：扩展初始化入口
- `app/api/`：健康检查、系统探针、设备、监测、风险结果、推理触发、告警等接口入口
- `app/service/`：设备、监测、风险结果、AI 推理编排、健康度与告警规则计算等业务层
- `app/repository/`：基于 mock 数据的数据访问层与后续真实数据源替换入口
- `app/schema/`：接口返回结构与 DTO 转换层
- `app/clients/`：后端调用外部 AI 服务的集成层
- `app/rules/`：健康度计算与告警级别映射等业务规则层
- `tests/`：最小自动化测试与业务接口测试
- `sql/mysql/ddl/`：MySQL 核心业务表 DDL
- `sql/influxdb/`：InfluxDB measurement 设计说明与样例

- 当前项目严格遵循“前后端分离 + AI 服务独立 + 双库协同（MySQL + InfluxDB）”的总体架构思路。
- MySQL 负责业务管理与关联型数据。
- InfluxDB 负责 ATP 高频监测时序数据。
- AI 服务当前采用独立 mock 推理服务形态，后端负责统一业务封装与健康度/告警解释规则。
- 当前阶段尚未接入真实神经网络推理、概率校准与真实不确定性估计，但已经完成从风险结果获取到健康度映射、告警级别生成的最小业务闭环。
- 后续开发将继续围绕论文定义的“状态感知 - 风险预测 - 健康评估 - 告警展示 - 运维管理”闭环逐步推进。
