## 开发原则

1. 每次只完成一个任务。
2. 每完成一个任务必须执行对应测试。
3. 不要一次性跨阶段大改。
4. 优先保证现有接口结构稳定。
5. 当前阶段允许继续使用 mock 数据。
6. 后续逐步替换为真实 MySQL、InfluxDB 和真实 AI 模型。
7. 前端页面风格必须保持和 Dashboard、设备台账一致。
8. 代码结构继续遵循当前分层方式：

```text
API -> Service -> Repository -> Schema -> Response
```

# 第六阶段：后端业务闭环补齐

## Task 27：告警状态更新接口

### 目标

补齐告警处理能力。

### 开发内容

新增接口：

```text
PATCH /api/v1/alerts/<alert_id>/status
```

请求体示例：

```json
{
  "alert_status": "HANDLED",
  "handler_id": 1,
  "handle_note": "已通知检修人员复核"
}
```

### 测试标准

- 合法状态更新成功。
- 告警不存在返回 404。
- 非法状态返回 400。
- 自动化测试覆盖成功、失败、非法参数。
